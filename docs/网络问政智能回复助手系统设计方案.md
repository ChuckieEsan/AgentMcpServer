# 网络问政智能回复助手系统设计方案

## 1. 引言

### 1.1 项目背景

12345政务服务热线是群众诉求表达的核心渠道，面临海量诉求人工处理效率低、跨部门协同难、民生热点预警滞后等痛点。本项目旨在构建一个智能回复助手，实现群众诉求的自动分类、智能回复与工单流转闭环，提升政务服务效率与群众满意度。

### 1.2 系统目标

- **智能优先**：基于历史案例、权责清单、部门办事清单、网络权威信息，对群众咨询类诉求实现高置信度自动回复。
- **人工兜底**：对无法自动回复的诉求自动创建工单，进入现有工单流转系统，由人工办理。
- **闭环迭代**：人工办结的优秀案例自动沉淀至知识库，持续优化系统回复效果。
- **安全合规**：全流程敏感信息脱敏、内容审核、权限管控，符合政务数据安全规范。

## 2. 系统总体架构

系统采用分层、模块化设计，自上而下分为五层，辅以两大支撑体系：

plaintext

```
┌─────────────────────────────────────────────────────────────┐
│                      交互层（群众端/政务端）                  │
├─────────────────────────────────────────────────────────────┤
│                    LangGraph Agent层                         │
│   （任务规划、工具调用、置信度评估、知识融合）                 │
├─────────────────────────────────────────────────────────────┤
│               MCP协议转换层（Spring Boot MCP Server）         │
│     （工具注册、路由代理、远程调用、服务发现集成）              │
├─────────────────────────────────────────────────────────────┤
│                   基础设施层（Higress + Nacos）               │
│          （统一入口、服务注册与发现、配置管理）                 │
├─────────────────────────────────────────────────────────────┤
│                  政务微服务层（现有业务服务）                   │
│   （权责清单、案例检索、部门知识库、工单系统、网络搜索等）       │
├─────────────────────────────────────────────────────────────┤
│                   知识库与数据层（向量库/关系库）               │
└─────────────────────────────────────────────────────────────┘
                        ▲                   ▲
                        │                   │
                ┌───────┴───────┐   ┌───────┴───────┐
                │ 安全合规体系   │   │ 监控运维体系   │
                └───────────────┘   └───────────────┘
```



### 2.1 各层职责

- **交互层**：群众通过小程序、Web等提交诉求，政务人员通过管理后台查看工单与统计。
- **LangGraph Agent层**：系统的智能中枢，负责任务规划、调用MCP工具、评估回复置信度、融合多源知识，决定直接回复或创建工单。
- **MCP协议转换层**：将Agent的MCP请求转换为对后端微服务的HTTP/gRPC调用，屏蔽服务细节。
- **基础设施层**：Higress作为统一网关处理认证、路由、限流；Nacos提供服务注册发现与动态配置。
- **政务微服务层**：现有业务服务，包括权责清单查询、历史案例检索、部门知识库、工单系统、网络搜索等。
- **知识库与数据层**：Milvus向量库存储历史案例、政策文档；关系库存储工单、用户信息等。

## 3. 模块详细设计

### 3.1 LangGraph Agent层

#### 3.1.1 核心职责

- 接收用户诉求文本。
- 调用权责清单工具获取匹配部门。
- 并行调用多个检索工具（历史案例、部门知识库、网络搜索）获取相关信息。
- 融合多源知识，生成回复候选。
- 评估回复置信度，若≥阈值则直接返回，否则调用工单创建工具。
- 维护对话状态（工单ID、处理进度等）。

#### 3.1.2 状态图设计

使用LangGraph的StateGraph实现有状态工作流，节点包括：

1. **诉求预处理**：文本清洗、敏感信息脱敏、核心要素提取。
2. **分类分拨**：查询权责清单，确定部门、类型、紧急程度。
3. **知识检索**：并行调用多个检索工具，返回结果列表。
4. **知识融合**：去重、排序、摘要，形成上下文。
5. **回复生成**：调用LLM生成回复文本。
6. **置信度评估**：调用评估服务，输出分数。
7. **分支决策**：根据分数选择“直接回复”或“创建工单”。
8. **工单创建**：调用工单系统API，返回工单号。
9. **结果输出**：向用户返回回复或工单号。

### 3.2 MCP协议转换层（Spring Boot MCP Server）

#### 3.2.1 设计目标

- 将LangGraph Agent的MCP调用转化为对后端微服务的实际调用。
- 与Nacos集成，动态发现后端服务。
- 支持工具映射配置，可热更新。

#### 3.2.2 核心组件

- **MCP端点**：使用Spring AI MCP Server Starter暴露SSE端点（如`/mcp`）。
- **工具注册管理器**：从Nacos配置中心加载工具映射表，为每个工具生成独立的Tool Bean（通过动态Bean注册），避免所有工具合并为一个。
- **远程调用代理**：使用Feign或WebClient，通过服务名从Nacos获取实例并调用。
- **配置监听**：监听Nacos配置变化，实时更新工具映射，无需重启。

#### 3.2.3 工具映射示例

yaml

```
tools:
  - name: "query_historical_cases"
    service: "case-retrieval-service"
    path: "/api/search"
    method: "POST"
    input_schema: {...}
  - name: "create_work_order"
    service: "workorder-service"
    path: "/api/orders"
    method: "POST"
```



### 3.3 基础设施层（Higress + Nacos）

#### 3.3.1 Higress网关

- **统一入口**：对外暴露MCP端点（如`https://api.gov.local/mcp`），所有Agent请求经过网关。
- **认证鉴权**：验证JWT或API Key，确保只有合法Agent访问。
- **服务发现集成**：配置Nacos为服务来源，动态获取MCP Server实例列表，实现负载均衡。
- **可观测性**：记录请求日志、耗时、状态码，输出到监控系统。

#### 3.3.2 Nacos

- **注册中心**：所有微服务（MCP Server、各政务微服务）启动时注册到Nacos，提供健康检查。
- **配置中心**：存储工具映射表、知识库地址、置信度阈值等配置，支持动态刷新。

### 3.4 政务微服务层

现有业务服务需封装为HTTP/gRPC接口，注册到Nacos。核心服务包括：

- **权责清单服务**：根据诉求文本匹配部门、事项、权责范围。
- **历史案例检索服务**：基于Milvus向量库，召回相似历史办结案例。
- **部门知识库服务**：提供各部门的办事指南、政策文件等。
- **网络搜索服务**：对接搜索引擎，返回网页结果，并进行权威性过滤。
- **工单系统服务**：提供创建工单、查询进度、接收办结事件的接口。
- **置信度评估服务**：接收诉求与回复，返回置信度分数。

### 3.5 知识库与数据层

- **Milvus向量库**：存储历史案例、政策法规、权责清单的向量嵌入，用于相似度检索。
- **关系数据库**：存储工单记录、用户信息、操作日志等结构化数据。
- **消息队列**：用于工单办结事件的异步通知（如Kafka），实现案例自动入库。

## 4. 核心机制设计

### 4.1 回复置信度评估

#### 4.1.1 评估维度

- **来源权威性**：根据知识来源元数据加权（政府官网1.0，历史案例0.8，普通网络0.3）。
- **完整性**：回复是否覆盖诉求关键要素（时间、地点、事件等）。
- **一致性**：回复与检索到的权威信息是否矛盾。
- **模型置信度**：生成回复的token级概率平均值。
- **规则检查**：是否包含含糊表述或敏感词。

#### 4.1.2 评估流程

Agent生成回复后，调用独立的置信度评估服务（可通过MCP工具封装），该服务返回0~1分数。若分数≥阈值（如0.8），直接返回；否则创建工单。

### 4.2 多渠道知识融合

#### 4.2.1 融合步骤

1. **并行检索**：Agent同时调用历史案例、部门知识库、网络搜索等工具。
2. **去重与排序**：对返回结果进行去重（向量相似度），并按“相关性×权威性”降序保留Top-K。
3. **摘要与组织**：对过长文本进行摘要，按类别组织成结构化提示词。
4. **生成回复**：将融合后的上下文与用户诉求输入LLM，生成回复。

#### 4.2.2 信息矛盾处理

若不同来源信息冲突，优先采用最新、最权威的来源，并在回复中标注依据，供用户核验。

### 4.3 网络搜索权威性限制

#### 4.3.1 多级过滤

- **域名白名单**：仅接受`.gov.cn`、`.org.cn`等可信域名，可通过搜索API限定`site:`参数。
- **内容质量评估**：对抓取的正文进行快速分析，判断是否包含政策文件编号、发布日期等特征。
- **缓存与预加载**：定期爬取权威网站，更新本地知识库，减少实时搜索依赖。

#### 4.3.2 实现方式

- 网络搜索工具内置多个搜索引擎备用，请求时附加白名单限制。
- 返回结果后，Agent根据URL域名决定是否抓取全文；抓取内容经质量评估后采纳。

### 4.4 与工单系统双向同步

#### 4.4.1 工单创建

Agent无法自动回复时，调用“创建工单工具”，传递诉求原文、分拨结果、置信度分数等，工单系统返回工单ID。Agent告知用户工单号。

#### 4.4.2 办结事件回传

- 工单系统办结后，发送消息至Kafka，包含工单ID、诉求原文、最终答复、办理部门等信息。
- **案例入库服务**监听消息，将最终答复文本分块、向量化后存入Milvus，标记为“人工办结案例”，并更新高频问答库。

#### 4.4.3 进度查询

Agent提供“查询工单进度”工具，用户可随时获取当前状态。

## 5. 业务流程示例

**场景**：群众通过小程序提交诉求：“请问灵活就业人员如何缴纳社保？”

1. **诉求受理**：LangGraph Agent接收文本，进行清洗、脱敏、要素提取（地点：默认本市，事件：灵活就业社保缴纳）。
2. **分类分拨**：调用权责清单工具，匹配到“人社局”，诉求类型为“咨询”。
3. **知识检索**：并行调用：
   - 历史案例检索：召回相似咨询案例3条。
   - 部门知识库：获取人社局最新的灵活就业参保指南。
   - 网络搜索：限定`.gov.cn`，检索到本市人社局官网发布的政策文件。
4. **知识融合**：去重后，将指南和案例摘要组织为上下文。
5. **回复生成**：LLM生成回复，包含参保条件、所需材料、办理渠道，并附上政策文件链接。
6. **置信度评估**：评估服务打分为0.95（高权威来源，完整覆盖诉求）。
7. **直接回复**：Agent通过接口将回复返回给用户。
8. **用户反馈**：用户可评价，评价数据用于后续优化。

若诉求为复杂投诉：“小区物业不作为，乱收费”，Agent分类后无法从知识库找到直接答案，置信度低（0.4），则调用工单创建工具，生成工单进入人工办理流程。

## 6. 安全与合规体系

### 6.1 数据安全

- **传输加密**：全链路HTTPS，敏感字段加密。
- **存储加密**：数据库敏感列使用国密算法加密。
- **脱敏展示**：前端默认脱敏身份证、手机号，仅授权人员可查看明文。

### 6.2 内容安全

- **前置拦截**：基于敏感词库过滤诉求和回复中的违规内容。
- **模型审核**：调用合规审核服务对生成内容进行二次校验。
- **人工兜底**：高风险内容自动转人工审核。

### 6.3 权限管控

- **RBAC**：按角色（超级管理员、政务办理员、普通用户）控制菜单和数据权限。
- **操作审计**：所有关键操作（工单创建、修改、查询）记录日志，留存≥1年。

## 7. 技术栈与部署方案

### 7.1 技术栈

| 组件          | 选型                        | 说明                       |
| :------------ | :-------------------------- | :------------------------- |
| Agent框架     | LangGraph                   | 支持有状态多智能体工作流   |
| MCP Server    | Spring Boot + Spring AI MCP | Java生态，集成Nacos、Feign |
| 网关          | Higress                     | 云原生，支持Nacos服务发现  |
| 注册/配置中心 | Nacos                       | 服务发现与动态配置管理     |
| 向量数据库    | Milvus                      | 高性能向量检索             |
| 关系数据库    | PostgreSQL / MySQL          | 存储工单等结构化数据       |
| 消息队列      | Kafka                       | 异步事件通知               |
| 部署          | Kubernetes + Docker         | 容器化编排，支持弹性伸缩   |

### 7.2 部署方案

- **开发测试环境**：Docker Compose部署所有服务，Nacos单节点，Milvus单机。
- **生产环境**：Kubernetes集群部署，Nacos集群，Milvus集群，Higress作为入口网关。MCP Server水平多实例，通过HPA自动扩缩容。

### 7.3 监控与运维

- **指标监控**：Prometheus采集服务指标，Grafana展示。
- **日志收集**：ELK集中存储日志。
- **链路追踪**：SkyWalking追踪跨服务调用链。

## 8. 总结

本系统以LangGraph Agent为核心，通过MCP协议标准化工具调用，结合Higress+Nacos构建云原生基础设施，实现了网络问政场景下的智能回复与人工兜底闭环。四大核心机制（置信度评估、知识融合、权威搜索、双向同步）确保了系统的准确性、安全性和可迭代性。该方案完全贴合政务需求，具备高可扩展性，可作为秋招项目亮点或企业落地方案。
